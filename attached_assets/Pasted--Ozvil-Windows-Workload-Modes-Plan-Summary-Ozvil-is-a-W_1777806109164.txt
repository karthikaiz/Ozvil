# Ozvil Windows Workload Modes Plan

## Summary

Ozvil is a Windows-first desktop utility that automatically prepares a PC for heavy creative, developer, gaming, and local AI workloads, then restores the system afterward.

The v1 product targets Windows users who feel real performance friction from background apps, cloud sync, browser memory pressure, power-plan throttling, laptop sleep, and noisy interruptions during heavy work. Ozvil detects selected heavy apps/processes, activates user-approved workload profiles, prevents sleep, applies Windows power settings where safe, monitors CPU/RAM/battery risk, pauses user-approved background apps, logs every action, and restores prior state when the session ends.

This file is the source of truth during implementation. Every phase must update status, test evidence, open risks, and completion notes before moving forward.

Technical feasibility references:

- Tauri v2 testing supports unit/integration tests with mock runtime: https://v2.tauri.app/develop/tests/
- Tauri v2 supports bundled sidecar binaries: https://tauri.app/develop/sidecar/
- Tauri v2 updater supports signed update artifacts and platform-specific releases: https://v2.tauri.app/ko/plugin/updater/
- Windows `powercfg` supports power scheme inspection and switching: https://learn.microsoft.com/en-us/windows-hardware/design/device-experiences/powercfg-command-line-options
- Windows `SetThreadExecutionState` can prevent sleep/display idle while an app is running: https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-setthreadexecutionstate

Fast-follow v1.1 candidates:

- Bandwidth/network warnings during heavy sessions.
- User-approved recovery command for suspected hung local AI/dev workflows.
- Community/shareable profile packs.
- Profile-specific bottleneck monitoring for disk I/O, GPU pressure, scratch disk pressure, and frame pacing where low-overhead Windows APIs allow it.
- Sequential profile dependency scripts, such as starting Docker containers before Build Mode and shutting them down during restore.
- Local webhook or Stream Deck integration for hardware control, after auth/security design.
- ASIO/audio-driver stream research after validating v1 Studio Mode behavior, with no custom virtual audio driver in scope.
- Advanced process priority tuning for explicitly approved apps, limited to safe levels such as Above Normal.

Deferred:

- Network/domain blocking, because it can conflict with VPNs, DNS, firewall rules, and user trust.
- Automatic restart on hang, because it can destroy work or hide real failures unless tightly constrained.
- UI automation or registry hacks for Windows Focus/notification control, because they are brittle across Windows updates.
- Exact VRAM/GPU allocation claims until low-overhead reliable metrics are validated across Windows GPU vendors and local AI backends.
- Local HTTP/webhook control endpoints without an authentication and local-only security model.
- macOS support, unless Windows v1 validates demand and cross-platform expansion becomes valuable.

## Application-Specific Processing Rule

Ozvil must evolve from generic CPU/RAM monitoring toward profile-specific bottleneck awareness. V1 may implement only safe, low-overhead subsets, but all future design should respect the workload architecture below.

- Render Mode: prioritize GPU compute, sustained disk I/O, external/cache drive availability, AC power, sleep prevention, and power throttling warnings.
- Build Mode: prioritize disk I/O conflicts, file locks, cloud-sync interference, RAM pressure, local AI stalls, and long-running process health.
- Design Mode: prioritize RAM pressure, scratch disk pressure, background disk spikes, and cloud-sync/file-lock risk.
- Game Mode Plus: prioritize frame pacing, background CPU wakeups, RAM pressure, and safe interruption reduction.
- Studio Mode: prioritize DAW/manual session protection, audio-driver activity where feasible, bursty CPU behavior, and avoiding false idle detection.
- Recording Mode: prioritize sleep prevention, interruption reduction, CPU/RAM pressure, and background app noise.

Design rule:

- Every new profile action or trigger must identify which workload bottleneck it protects.
- Deep telemetry such as GPU, ASIO, PCIe, external drive sleep, frame pacing, and scratch disk monitoring is future scope unless a low-overhead Windows API is validated.
- VRAM/GPU metrics are experimental until reliable low-overhead Windows APIs are validated across NVIDIA, AMD, Intel, browser-hosted local AI, Python backends, and WSL-style workflows.
- ASIO activity cannot be assumed visible through WASAPI; true ASIO detection remains research-only and must not require a virtual audio driver in v1/v1.1.
- Generic CPU/RAM thresholds are fallback signals, not the long-term product moat.

## Built-In Profiles

### Render Mode

Audience:

- Video editors
- Animators
- 3D artists
- Colorists

Default triggers:

- App/process running: DaVinci Resolve, Premiere Pro, After Effects, Blender, Cinema 4D, Media Encoder
- Optional process CPU threshold above 60 percent for 120 seconds
- Manual CLI/UI start

Default actions:

- Prevent sleep
- Switch to approved high-performance power plan where configured
- Watch battery and AC power
- Watch RAM pressure
- Watch CPU pressure
- Watch disk I/O and GPU pressure when low-overhead metrics are validated
- Warn about cache/external drive risk when the render workflow depends on removable or external storage
- Pause user-approved cloud-sync/background apps
- Restore on app quit, resource idle, or manual stop

Audience:

- Music producers
- Podcasters
- Audio engineers

- Burst-aware CPU/audio activity as a supporting signal, not the only trigger

Default actions:

- Prevent sleep

- Pause user-approved background apps

## Core Interfaces

Frontend-facing profile model:

```ts
type Profile = {
  id: string;
  name: string;
  modeType: "render" | "studio" | "build" | "game" | "design" | "recording" | "custom";
  triggers: Trigger[];
  actions: Action[];
  restorePolicy: "on_app_quit" | "on_resource_idle" | "manual";
  approvalMode: "ask_first" | "automatic_after_trusted";
  enabled: boolean;
};

type Trigger =
  | { kind: "app_running"; appId: string }
  | { kind: "process_running"; processName: string }
  | { kind: "cpu_above"; processName?: string; percent: number; durationSeconds: number }
  | { kind: "memory_above"; processName?: string; mb: number; durationSeconds: number }
  | { kind: "manual_cli"; profileId: string }
  | { kind: "manual_ui"; profileId: string };

type Action =
  | { kind: "prevent_sleep" }
  | { kind: "reduce_interruptions" }
  | { kind: "set_power_plan"; planId: string }
  | { kind: "pause_approved_app"; appId: string }
  | { kind: "watch_battery"; warnBelowPercent: number }
  | { kind: "watch_memory"; warnAbovePercent: number }
  | { kind: "watch_cpu"; warnAbovePercent: number }
  | { kind: "run_approved_script"; scriptId: string };
```

Rust Windows adapter contract:

```rust
trait WindowsAdapter {
    fn list_processes(&self) -> Result<Vec<ProcessInfo>>;
    fn read_system_status(&self) -> Result<SystemStatus>;
    fn snapshot_state(&self, actions: &[Action]) -> Result<SystemSnapshot>;
    fn apply_action(&self, action: &Action) -> Result<ActionResult>;
    fn restore_snapshot(&self, snapshot: &SystemSnapshot) -> Result<RestoreResult>;
}
```

Safe Mode behavior:

- `ozvil --safe-mode` and `ozvil safe-start` launch Ozvil with all automation disabled.
- The Windows installer must create a dedicated Start Menu shortcut named `Ozvil (Safe Mode)` that passes the safe-mode flag.
- Holding Shift during UI startup may request Safe Mode where technically feasible, but it is only a convenience path, not the primary safety hatch.
- Safe Mode must allow profile editing/deletion, log export, manual restore, and Global Pause control.
- Safe Mode must not run profiles, scripts, app pauses, or power actions on startup.

Conflict defaults:

- Emergency restore can always run, even when Global Pause is active.
- Global Pause blocks all new automatic activations.
- Manual activation while Global Pause is active requires explicit confirmation or a temporary pause override.
- Existing active sessions continue unless the user chooses "pause and restore now."
- Manual CLI/UI activation is sticky and beats automatic activation.
- Automatic profiles cannot replace an active manual profile; Ozvil logs the match and may suggest switching.
- Recording and Studio modes beat other automatic profiles because interruption risk is highest.
- Foreground app profile beats background app profile where foreground detection is available.
- More recently activated foreground app wins remaining ties.
- If still tied, keep current active profile and log the suppressed match.

Test gate:

- Review each built-in profile against at least one realistic Windows workflow.

Exit criteria:

- V1 behavior is clear enough to implement without product decisions.

## Security And Safety Principles

- No hidden system mutations.
- No background app control without explicit approval.
- No arbitrary script execution without explicit approval.
- No approved script runs without dry-run visibility, timeout handling, and activity logging.
- No storing secrets in plain text.
- No default telemetry.
- No destructive system commands.
- No registry cleaning.
- No undocumented registry hacks for notification/Focus control.
- No UI automation for notification/Focus control in v1.
- No kernel drivers in v1.
- No virtual audio drivers in v1 or v1.1.
- No local webhook/control server in v1.
- Prefer non-persistent OS controls over persistent system setting changes.
- Sleep prevention should use non-persistent Windows execution-state/power APIs where available.
- Sleep prevention is treated as requested/monitored, not guaranteed, because Battery Saver/OEM policies can override it.
- Power-plan switching must be capability-detected because Modern Standby/OEM devices may hide or disable traditional plans.
- Ozvil must reconcile stale sessions on startup after crash, force-quit, or reboot.
- Direct Windows Focus/notification control is best-effort only and must fail visibly if unsupported.
- Application-specific bottleneck processing is a long-term design rule; generic CPU/RAM monitoring is not enough for the product vision.
- ASIO/WASAPI audio activity detection is research-only until proven feasible without driver-level components.
- Safe Mode must disable automation before any profile, script, app pause, or power action can run.
- The dedicated `Ozvil (Safe Mode)` shortcut is the primary GUI safety hatch; Shift startup is only best-effort.
- Activity logs must be retained, rate-limited, and pruned so SQLite cannot grow unbounded.
- VRAM/GPU monitoring must remain experimental until reliable low-overhead vendor-neutral behavior is validated.
- Every privileged action must have a clear user-facing reason.
- Every applied action must have a restore path or a clear reason why restore is not possible.
- Network/domain blocking is deferred until it has a reliable permissions, restore, and support story.
- Automatic process restart is deferred unless explicitly configured by the user as a recovery command.
- Process priority tuning is deferred to an advanced, explicit-approval feature and must never use High or Realtime by default.
