# Building Ozvil

## Prerequisites

- **Windows 10 or 11** (required — Windows-native APIs are used)
- **Rust** stable toolchain: https://rustup.rs
  ```
  rustup target add x86_64-pc-windows-msvc
  ```
- **Node.js** 20+ and **pnpm** 9+:
  ```
  npm install -g pnpm
  ```
- **Tauri v2 CLI**:
  ```
  pnpm add -g @tauri-apps/cli@^2
  ```
- **WebView2** (pre-installed on Windows 11; download for Windows 10):
  https://developer.microsoft.com/microsoft-edge/webview2/
- **WiX Toolset v4** (for MSI builds):
  https://wixtoolset.org/docs/intro/
- **NSIS 3.x** (for EXE installer builds):
  https://nsis.sourceforge.io/Download
- **Windows SDK** (for `signtool.exe`, optional — only needed for code signing):
  https://developer.microsoft.com/windows/downloads/windows-sdk/

## Production Build

### Quick (uses helper script)

```powershell
cd ozvil
.\scripts\build-release.ps1              # build only
.\scripts\build-release.ps1 -SkipTests  # skip Rust tests
.\scripts\build-release.ps1 -Sign -Thumbprint "YOUR_CERT_THUMBPRINT"  # build + sign
```

### Manual

```bash
cd ozvil
pnpm tauri build --target x86_64-pc-windows-msvc --bundles nsis,msi
```

## Installer Details

### Start Menu shortcuts

Both installers create two Start Menu shortcuts under `Programs\Ozvil\`:

| Shortcut | Arguments | Purpose |
|---|---|---|
| **Ozvil** | *(none)* | Normal launch |
| **Ozvil (Safe Mode)** | `--safe-mode` | Primary safety hatch — automation fully disabled |

**WiX (MSI):** Safe Mode shortcut defined in `src-tauri/wix/safe_mode_shortcut.wxs`
and referenced via `tauri.conf.json > bundle.windows.wix.fragmentPaths`.

**NSIS (EXE):** Safe Mode shortcut created by the `!customInstall` macro in
`src-tauri/nsis/installer.nsi` and removed by `!customUninstall` on uninstall.

### Verify after installing

```powershell
.\scripts\verify-installer.ps1
```

Checks: executable present, both shortcuts exist, Safe Mode shortcut passes
`--safe-mode`, Authenticode signature valid (if signtool available), CLI smoke tests.

## Code Signing

See **SIGNING.md** for the full workflow. Quick reference:

```powershell
# Generate updater key pair (one time only)
.\scripts\generate-keys.ps1

# Sign already-built artifacts
.\scripts\sign-release.ps1 -Thumbprint "YOUR_CERT_THUMBPRINT"
.\scripts\sign-release.ps1 -PfxPath ".\ozvil.pfx" -PfxPassword "password"

# Dry run (validate without signing)
.\scripts\sign-release.ps1 -Thumbprint "..." -DryRun
```

## Auto-Updater

Ozvil uses the Tauri updater plugin. When the app launches (non-Safe Mode only),
it calls:

```
GET https://update.ozvil.app/windows/x86_64/<current_version>
```

- **204** → no update available (silent)
- **200** → update available → app shows update dialog

The update payload must be signed with the ed25519 private key generated by
`scripts/generate-keys.ps1`. The matching public key must be in
`tauri.conf.json > plugins.updater.pubkey`.

See `update-server/README.md` for the response format and a minimal Node.js
server example.

**Frontend events emitted by the updater:**

| Event | Payload | When |
|---|---|---|
| `update-available` | `{ version, notes }` | Update found on startup |
| `update-download-progress` | `{ chunkLength, contentLength }` | During download |
| `update-install-ready` | *(none)* | Download complete, ready to install |

**Tauri commands for manual update flow:**

| Command | Returns |
|---|---|
| `check_update_command` | `UpdateInfo \| null` |
| `install_update_command` | `void` (installs and relaunches) |

## CI / CD (GitHub Actions)

### Continuous Integration

`.github/workflows/ci.yml` — runs on every push to `main` / `develop`:
1. TypeScript typecheck (`pnpm tsc --noEmit`)
2. `cargo fmt --check`
3. `cargo clippy -- -D warnings`
4. `cargo test --all`

### Release Pipeline

`.github/workflows/release.yml` — triggered by version tags (e.g. `v0.1.0`):
1. Runs all CI checks
2. Imports code-signing certificate from GitHub secrets
3. Runs `pnpm tauri build --bundles nsis,msi` (Authenticode signed)
4. Tauri automatically signs `.sig` files with `TAURI_SIGNING_PRIVATE_KEY`
5. Uploads all artifacts + `.sig` files to a draft GitHub Release

**Required GitHub Actions secrets:**

| Secret | Value |
|---|---|
| `TAURI_SIGNING_PRIVATE_KEY` | Contents of `ozvil-updater.key` (from `scripts/generate-keys.ps1`) |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | Password chosen during key generation |
| `WINDOWS_CERTIFICATE` | Base64-encoded `.pfx` code-signing certificate |
| `WINDOWS_CERTIFICATE_PASSWORD` | Password for the `.pfx` file |

To create a release:

```bash
git tag v0.1.0
git push origin v0.1.0
```

## CLI Usage

After building, the same `ozvil.exe` binary functions as a CLI:

```
ozvil status                               # human-readable status
ozvil status --agent                       # stable minimal JSON
ozvil --safe-mode                          # launch GUI with automation disabled
ozvil safe-start                           # same as --safe-mode

ozvil profiles list                        # list all profiles
ozvil start render                         # activate Render Mode
ozvil start "my custom profile"            # activate by name
ozvil dry-run render                       # preview without applying
ozvil stop                                 # end active session + restore
ozvil restore                              # restore without stopping session

ozvil logs export --format json            # export to stdout
ozvil logs export --format csv --output logs.csv

ozvil profile export "Build Mode" --output build-mode.json
ozvil profile import my-profile.json
```

## Safe Mode

Launch with all automation disabled:

```
ozvil --safe-mode
```

Or use the **"Ozvil (Safe Mode)"** Start Menu shortcut (created by the installer).

In Safe Mode:
- All automatic profile activation is suppressed
- No scripts, process pauses, sleep prevention, or power plan changes run
- Profile editing, log export, manual snapshot restore, and Global Pause still work
- The updater does not run (no background tasks at all)
- The UI shows a persistent Safe Mode banner

## Security Principles

Per spec — Ozvil will not:
- Perform hidden system mutations
- Control apps without explicit user approval (trust model required)
- Execute scripts without approval + timeout + activity log entry
- Touch the registry except for approved Start Menu shortcut markers
- Enable default telemetry
- Use kernel drivers
- Implement undocumented API hacks (Focus Assist returns `UnsupportedCapability`)

Sleep prevention uses non-persistent `ES_CONTINUOUS | ES_SYSTEM_REQUIRED`
(auto-released on process exit). Power plan switching checks capability first.
